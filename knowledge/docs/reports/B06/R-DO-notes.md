### 1. Infrastructure for Optimization Workloads

Optimization workloads have distinct infrastructure requirements depending on the solver type. Classical solvers like Gurobi and OR-Tools are CPU-bound and benefit from high-core-count machines. A Gurobi MIP solver scales nearly linearly up to 8-16 cores for parallel branch-and-bound, making instances like AWS c7i.8xlarge (32 vCPU) or Azure Fsv2-series ideal. Memory is the second constraint: large MIP models with hundreds of thousands of variables can consume 16-64 GB of RAM, so memory-optimized instances (r-series) are needed for the biggest problems. Neural combinatorial optimization solvers (using reinforcement learning to solve routing or scheduling) require GPUs, typically T4 or A10G for inference. The infrastructure should separate these workload types into distinct node pools in Kubernetes, with appropriate taints and tolerations so that GPU-hungry neural solvers do not compete with CPU-bound classical solvers for scheduling priority.

### 2. Containerization for Solvers

Packaging solvers into Docker containers requires careful handling of commercial licenses. For Gurobi, the container must either mount a license file at runtime or connect to a Gurobi token server running on the host network. The Dockerfile uses a multi-stage build: the first stage installs gurobipy and OR-Tools into a Python virtual environment, and the second stage copies only the venv into a slim Debian base image. The final image is approximately 800 MB with Gurobi or 400 MB with only open-source solvers. OR-Tools is straightforward to containerize as it installs via pip with no external dependencies. For reproducibility, solver versions are pinned exactly (e.g., gurobipy==11.0.3, ortools==9.10.4067). A health check endpoint verifies that the solver can load and solve a trivial 2-variable LP, ensuring the license is valid and the binary is functional. Images are stored in a private container registry with vulnerability scanning enabled.

### 3. Scaling Solver Workers

Solver workers are scaled horizontally using Celery with Redis as the broker. A KEDA (Kubernetes Event Driven Autoscaler) scaler monitors Redis queue depth and spins up additional solver pods when the pending job count exceeds a threshold (e.g., more than 5 pending jobs per available worker). Scale-down is slower, with a 10-minute cooldown to avoid thrashing. For batch optimization windows (e.g., nightly route planning), a scheduled HPA override pre-scales the worker pool to a known size 15 minutes before the batch window opens. Preemptible or spot instances are used for batch workloads, with Celery's acks_late setting ensuring that a preempted job is automatically retried on another worker. Each worker enforces a per-job time limit using the solver's native timeout parameter plus a hard process-level timeout as a safety net. Worker pods request specific CPU and memory based on the expected problem size tier.

### 4. CI/CD for Optimization Models

The CI/CD pipeline for optimization code includes specialized stages beyond standard software testing. The first stage runs constraint validation unit tests that verify each business rule is correctly encoded as a mathematical constraint. The second stage solves a suite of benchmark problems (drawn from TSPLIB, CVRPLIB, or custom internal benchmarks) and compares solution quality and runtime against stored baselines. A solution quality gate fails the build if the objective value degrades by more than 1% on any benchmark or if runtime increases by more than 20%. The pipeline also runs a "canary solve" that submits a representative production-scale problem to a staging solver and validates the result end-to-end. Deployment uses blue-green strategy: the new solver version runs alongside the old one, processing a fraction of traffic, with automated rollback if solution quality metrics degrade. Model files (MPS/LP exports) from failed test cases are archived as artifacts for debugging.

### 5. Monitoring Optimization Systems

Monitoring optimization systems requires domain-specific metrics beyond standard application monitoring. Key metrics include: solver runtime per job (p50, p95, p99), optimality gap at termination (0% means proven optimal), number of infeasible problems per hour (spikes indicate data quality issues), queue depth and wait time (capacity planning signals), and business KPIs like total route distance or schedule makespan. These metrics are exported via Prometheus and visualized in Grafana dashboards organized by problem type and tenant. Alerts fire when the p95 solver runtime exceeds the SLA threshold, when the infeasibility rate exceeds 5%, or when the average optimality gap exceeds 2%. A weekly report summarizes optimization cost savings: comparing the optimized solution cost against a naive baseline (e.g., nearest-neighbor routing) to quantify the value delivered by the solver. Solver logs are shipped to Elasticsearch for root-cause analysis of failures.

### 6. Solver Licensing & Cost

Solver choice has major cost implications. Gurobi charges approximately $12,000-15,000 per year for a single named-user commercial license, or significantly more for compute server licenses that support concurrent solves across a cluster. IBM CPLEX has similar pricing through the IBM Decision Optimization suite. For organizations that cannot justify these costs, free alternatives include: Google OR-Tools (Apache 2.0, excellent for CP and routing), HiGHS (MIT license, competitive LP/MIP solver), SCIP (academic-free, strong MIP), and CBC (Coin-OR, mature LP/MIP solver). In benchmarks, Gurobi typically solves MIP problems 2-10x faster than free alternatives, but for many practical problems the gap is small enough that free solvers suffice. A pragmatic approach is to develop against OR-Tools and HiGHS, benchmark production problems, and only invest in Gurobi if the free solvers cannot meet SLA requirements. For startups, Gurobi offers free academic and evaluation licenses.

### 7. Cost Optimization for Solver Infra

Solver infrastructure costs can be controlled through several strategies. First, use spot or preemptible instances for batch optimization workloads; since jobs are idempotent and retryable, a 60-70% cost reduction is achievable. Second, right-size solver VMs: profile actual CPU and memory usage and select instance types that match. Many teams over-provision memory when the bottleneck is CPU. Third, tune solver parameters for faster convergence: setting an appropriate MIP gap tolerance (e.g., 0.5% instead of the default 0.01%) can reduce solve time by 80% with negligible solution quality impact for business applications. Fourth, implement solution caching to avoid re-solving identical or near-identical problems. Fifth, use tiered compute: solve small problems on cheap shared instances and only dispatch to expensive high-core machines for large problems. Sixth, schedule batch workloads during off-peak hours when spot prices are lowest. A well-optimized setup can reduce monthly solver infrastructure costs from thousands to hundreds of dollars for mid-scale deployments.

### 8. Vietnamese Deployment

Deploying optimization systems in Vietnam requires adapting to local infrastructure and regulatory considerations. For manufacturing companies (textiles, electronics assembly) that handle sensitive production data, on-premise solver deployment is preferred. A compact deployment runs Gurobi or OR-Tools on a single high-core server (Dell PowerEdge or similar) behind the factory firewall, with results pushed to a local dashboard. For logistics startups in Ho Chi Minh City and Hanoi, cloud deployment on AWS ap-southeast-1 (Singapore) or Google Cloud asia-southeast1 provides the lowest latency from Vietnam. Vietnamese universities (Bach Khoa, FPT University, VNU) can obtain free Gurobi academic licenses, making it an accessible tool for operations research education. Network reliability varies outside major cities, so optimization systems serving rural delivery networks should support offline mode: pre-compute solutions during connectivity windows and cache them locally. Data residency requirements for government-related optimization projects (public transit scheduling, utility network planning) may mandate hosting within Vietnam, where local providers like Viettel IDC or CMC Cloud offer suitable infrastructure.
