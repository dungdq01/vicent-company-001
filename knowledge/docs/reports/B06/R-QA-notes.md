# QA Engineering Notes: B06 Optimization
## By R-QA — Date: 2026-03-31

### 1. Testing Strategy for Optimization Systems

Testing optimization systems requires a layered approach that validates correctness at every level. Unit tests verify that individual constraints are formulated correctly: given a known set of inputs, does the generated mathematical model contain the expected coefficients, bounds, and constraint types? Integration tests validate the full pipeline from API request through solver execution to result delivery, ensuring that the async job submission, solver invocation, result storage, and notification chain all work together. Regression tests maintain a suite of benchmark problems with known optimal or best-known solutions, and every code change must produce solutions within an acceptable tolerance of these baselines. End-to-end tests simulate production scenarios: submit a realistic problem via the public API, wait for the async result, and validate the solution against business rules. The test pyramid for optimization is bottom-heavy on constraint unit tests (fast, hundreds of them) with fewer but critical integration and regression tests. Test execution time is managed by using small problem instances for unit and integration tests, reserving full-scale benchmarks for nightly CI runs.

### 2. Solution Quality Validation

Every solution returned by the optimization system must pass quality validation before reaching the client. The first check is feasibility: does the solution satisfy every constraint in the original problem? This is verified by substituting the decision variable values back into each constraint and checking that no violation exceeds a numerical tolerance (typically 1e-6). The second check is the optimality gap: for MIP solvers, this is the difference between the best solution found and the best theoretical bound, expressed as a percentage. A gap of 0% means the solution is proven optimal; gaps under 1% are typically acceptable for business applications. The third check is dominance: does the solution outperform simple heuristic baselines? If the "optimized" route is longer than a nearest-neighbor heuristic, something is wrong. The fourth check is business rule compliance: constraints encode business rules, but some rules are better validated independently. For example, verify that no driver exceeds their maximum working hours, that all deliveries fall within time windows, and that vehicle capacities are respected. These checks run automatically in the result pipeline and flag any solution that fails.

### 3. Benchmark Problem Testing

The optimization community maintains extensive benchmark libraries that provide known-optimal or best-known solutions for standard problem types. TSPLIB contains Traveling Salesman Problem instances ranging from 14 to 85,900 cities with proven optimal solutions for many. CVRPLIB provides Capacitated Vehicle Routing Problem instances with best-known solutions and proven lower bounds. MIPLIB contains general Mixed-Integer Programming instances used to benchmark solvers. The QA suite includes a representative sample from each library, selected to cover different problem sizes and difficulty levels. For each benchmark, the test records: solution objective value, optimality gap, solve time, and memory usage. Results are compared against both the known optimal and the previous version's performance. A dashboard tracks benchmark performance over time, making it easy to spot regressions when solver versions are upgraded or constraint formulations are modified. Custom internal benchmarks are created from anonymized production data to ensure testing reflects real-world problem characteristics.

### 4. Constraint Correctness Testing

Constraints are the heart of any optimization model, and incorrect constraints produce silently wrong solutions. Each business rule must be tested independently. For a vehicle routing problem, test that: capacity constraints prevent overloading (submit a problem where all orders exceed vehicle capacity and verify infeasibility), time window constraints reject late arrivals (fix a route that would arrive late and verify the constraint is violated), and precedence constraints enforce pickup-before-delivery ordering. Boundary condition tests are critical: what happens when a time window has zero width (exact appointment time), when demand exactly equals capacity, or when the distance between two points is zero? Constraint conflict detection tests submit problems with intentionally contradictory constraints and verify that the solver correctly reports infeasibility and the IIS analysis identifies the conflicting constraint set. Property-based testing generates random problem instances and verifies that solutions always satisfy invariants: total demand served equals total demand requested, no vehicle visits the same location twice, and route costs are non-negative.

### 5. Performance Testing

Performance testing for optimization systems measures both solver runtime and system throughput. Scaling tests solve the same problem type at increasing sizes: 100 variables, 1,000 variables, 10,000 variables, and 100,000 variables, recording solve time and memory consumption at each level. This establishes the empirical scaling curve and identifies the problem size at which the solver hits its practical limit. Solver time limit tests verify that the timeout mechanism works correctly: submit a problem known to take 10 minutes and set a 5-second time limit, then verify that the solver returns the best feasible solution found within the limit rather than hanging or crashing. Concurrent load tests simulate multiple tenants submitting optimization jobs simultaneously to verify that the queuing system distributes work fairly and that solver workers do not interfere with each other (especially important for Gurobi license token management). Memory profiling identifies leaks in the model construction code, which is common when solver objects are not properly disposed of between runs. Performance baselines are stored and compared in CI to catch regressions early.

### 6. Edge Case Testing

Optimization systems encounter a variety of edge cases that must be handled gracefully. Empty input: submitting a problem with zero orders or zero vehicles should return an empty solution or a clear error, not crash the solver. Infeasible problems: when no solution satisfies all constraints, the system must detect infeasibility and return a meaningful explanation rather than timing out. Unbounded problems: if the objective can improve infinitely (usually a modeling error, such as a missing non-negativity constraint), the solver should report this specifically rather than returning an arbitrarily large solution. Degenerate cases: a routing problem with one order and one vehicle, a scheduling problem with one task, or a problem where all costs are identical should all produce correct trivial solutions. Numerical precision issues arise when constraint coefficients span many orders of magnitude (e.g., mixing dollar costs with millisecond times); tests verify that the solver handles these without numerical instability. Duplicate inputs (identical orders, identical locations) must not cause crashes or incorrect constraint counting. Extremely large coefficients or very small epsilon values are tested to ensure the solver's numerical tolerances are appropriate.

### 7. Regression Testing for Solver Updates

Solver upgrades (e.g., Gurobi 10 to 11, OR-Tools 9.9 to 9.10) can change solution behavior even when the API is identical. A new solver version may find better solutions for some problems but worse for others due to changes in heuristics, cut generation, or presolve techniques. The regression test protocol runs the full benchmark suite on both the old and new solver versions and produces a comparison report. For each benchmark instance, the report shows: old vs new objective value, old vs new solve time, and whether optimality was proven. Acceptable criteria: no benchmark should degrade by more than 0.5% in objective value, and no more than 10% of benchmarks should show runtime increases exceeding 50%. If these thresholds are violated, the upgrade is blocked and the specific problem instances are investigated. The test suite also verifies API compatibility: parameter names, solution format, and error codes should remain consistent. Solver changelogs are reviewed for deprecated features or changed default parameter values that might affect production behavior.

### 8. Production Monitoring Checklist

Continuous monitoring in production catches issues that testing alone cannot anticipate. The checklist includes: solver failure rate (any solve that crashes rather than returning infeasible or timeout), tracked per solver type and problem type, with an alert threshold of more than 1% failure rate. Constraint violation rate: even if a solver reports a feasible solution, an independent validator checks all constraints, and any violation triggers an alert. Optimality gap distribution: track the histogram of gaps across all production solves; a shift toward larger gaps indicates that problem sizes are growing or solver performance is degrading. Solve time SLA compliance: what percentage of jobs complete within the contracted time limit, broken down by tenant and priority tier. Business KPI drift: track the average cost, distance, or makespan of optimized solutions over time; a gradual increase may indicate changes in input data characteristics that require model updates. Queue health: pending job count, average wait time, and worker utilization. Resource utilization: CPU, memory, and solver license token usage across the worker pool. All metrics feed into a single Grafana dashboard with daily and weekly summary reports sent to stakeholders.
